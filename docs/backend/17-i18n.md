# 17 — Système i18n (traductions)

## Concept

Système de traduction **entièrement custom** — pas de `gettext` / fichiers `.po` Django.

Deux approches complémentaires selon le type de contenu :

| Type de contenu | Approche | Exemple |
|---|---|---|
| Contenu géré par l'admin (FAQ, règles, légal…) | Champs suffixés `_fr` / `_en` sur le modèle | `FaqItem.question_fr`, `FaqItem.question_en` |
| Strings système (erreurs, SMS, notifications…) | `TranslationService` JSON | `translate("errors", "otp_invalid", language)` |

La clé de lecture : si un humain écrit le texte via Django Admin → suffixes sur le modèle. Si le code génère le texte → JSON i18n.

Deux langues supportées : **`fr`** (défaut — langue principale de RunDate) et **`en`**.

## Format des fichiers JSON

Chaque fichier représente une **section** (thème). Structure toujours identique :

```json
{
    "fr": {
        "otp_sent": "Ton code a été envoyé au {phone}.",
        "otp_invalid": "Code invalide ou expiré. Réessaie.",
        "event_confirmed": "Ton run du {date} à {neighborhood} est confirmé! 🎉",
        "event_cancelled_no_quorum": "Run annulé — pas assez de coureurs cette fois.",
        "token_expired": "Ta session a expiré. Reconnecte-toi.",
        "application_unauthorized": "Application non autorisée.",
        "database_error": "Une erreur est survenue. Réessaie plus tard.",
        "event_full": "Cet événement est complet.",
        "bio_generation_failed": "Impossible de générer ta bio pour le moment. Réessaie plus tard."
    },
    "en": {
        "otp_sent": "Your code has been sent to {phone}.",
        "otp_invalid": "Invalid or expired code. Please try again.",
        "event_confirmed": "Your run on {date} in {neighborhood} is confirmed! 🎉",
        "event_cancelled_no_quorum": "Run cancelled — not enough runners this time.",
        "token_expired": "Your session has expired. Please log in again.",
        "application_unauthorized": "Application not authorized.",
        "database_error": "Something went wrong. Please try again later.",
        "event_full": "This event is full.",
        "bio_generation_failed": "We couldn't generate your bio right now. Please try again later."
    }
}
```

Les textes supportent le formatage Python avec `{variable}` pour les valeurs dynamiques.

Les variantes régionales (`fr-CA`, `en-US`) sont automatiquement normalisées vers `fr` ou `en`.

## `TranslationService` — le moteur central

Classe stateless avec cache de classe (`_cache: Dict`). Les fichiers JSON sont lus **une seule fois** au premier appel, puis mis en mémoire pour toute la durée de vie du processus.

```python
# apps/i18n/services.py
class TranslationService:
    _cache: Dict[str, Dict] = {}

    @classmethod
    def get_translation(cls, section: str, identifier: str, language: str, **kwargs) -> str:
        """Retrieve a translation with optional variable substitution."""
        data = cls._load_section(section)
        lang = cls._normalize_language(language)
        text = data.get(lang, {}).get(identifier)
        if text is None:
            return f"*[{section}.{identifier}:{lang}]*"   # jamais silencieux
        return text.format(**kwargs) if kwargs else text

    @classmethod
    def _normalize_language(cls, language: str) -> str:
        """Normalize fr-CA → fr, en-US → en, etc."""
        lang = (language or 'fr').lower().split('-')[0].split('_')[0]
        return lang if lang in ('fr', 'en') else 'fr'
```

### Méthodes disponibles

| Méthode | Usage |
|---|---|
| `get_translation(section, identifier, language, **kwargs)` | Récupère une traduction avec variables optionnelles |
| `translate_with_context(section, identifier, context_obj)` | Détecte la langue depuis un objet |
| `get_missing_translations(section, reference_language="fr")` | Trouve les clés manquantes |
| `clear_cache()` | Vide le cache (développement) |
| `reload_section(section)` | Recharge un fichier JSON sans redémarrer |

**Comportement si une clé est manquante** : retourne `*[section.identifier:lang]*` — visible immédiatement dans l'interface, jamais une erreur silencieuse.

## Points d'entrée Python

### 1. Import direct — le plus courant dans les mutations GraphQL

```python
from apps.i18n.utils import translate

# Dans une mutation (language vient du décorateur @extract_language)
translate("errors", "token_expired", language)

# Avec variables dynamiques
translate("notifications", "event_confirmed", language,
          date="samedi 28 mars", neighborhood="Plateau")
```

### 2. Alias court `t`

```python
from apps.i18n.services import t

t("errors", "otp_invalid", language)
```

### 3. Avec contexte — détection automatique de langue depuis un objet

```python
from apps.i18n.utils import tc

# Détecte la langue depuis obj.language, obj.get_user_language()
# ou obj.user_access.language
tc("notifications", "event_confirmed", user_access,
   date="samedi 28 mars", neighborhood="Plateau")
```

## Intégration avec les mutations GraphQL

La langue arrive via le header `language`, injectée par `@extract_language` :

```python
@log_graphql_request
@require_authorization
@require_user_access_token
@extract_language         # ← injecte `language` depuis le header HTTP
def mutate(self, info, application_access, user_access, language, ...):
    if not phone:
        return AccountsUserAccessRequestOtpPayload(
            status=False,
            errors={"phone": translate("errors", "phone_is_required", language)},
        )

    try:
        services.request_otp(phone)
        return AccountsUserAccessRequestOtpPayload(status=True, errors={})
    except OtpRateLimitError:
        return AccountsUserAccessRequestOtpPayload(
            status=False,
            errors={"phone": translate("errors", "otp_rate_limit_exceeded", language)},
        )
```

## Intégration avec les notifications

Les management commands et signals utilisent la langue de l'utilisateur destinataire :

```python
# apps/notifications/services.py
def create_event_confirmed_notification(user, event):
    language = user.user_accesses.first().language if ... else 'fr'
    AppNotification.objects.create(
        user=user,
        type=NotificationType.RUN_CONFIRMED,
        title=translate("notifications", "run_confirmed_title", language),
        body=translate("notifications", "run_confirmed_body", language,
                       date=event.date.strftime("%A %d %B"),
                       neighborhood=event.neighborhood.name),
    )
```

## Contenu admin — suffixes `_fr` / `_en` sur les modèles

### Convention

Tout champ texte d'un modèle géré via Django Admin doit avoir des variantes linguistiques par suffixe :

```python
# ✅ Convention — suffixes obligatoires
class FaqItem(BaseModel):
    question_fr = TextField()
    question_en = TextField(blank=True)
    answer_fr   = TextField()
    answer_en   = TextField(blank=True)

# ❌ Jamais un champ `language` séparé pour du contenu admin
class FaqItem(BaseModel):
    question = TextField()
    language = CharField(choices=[("fr", "fr"), ("en", "en")])  # NON
```

### Règles

- `_fr` est **obligatoire** — jamais vide en prod
- `_en` peut être vide en développement, mais doit être rempli avant mise en production
- Pas de champ `language` FK ou CharField sur les modèles de contenu — une seule ligne = toutes les langues
- Pour ajouter une nouvelle langue future (ex: `_es`) : chercher `_fr` dans le code, ajouter `_es` partout

### Modèles avec suffixes `_fr` / `_en`

| Modèle | Champs suffixés |
|---|---|
| `FaqItem` | `question_fr/en`, `answer_fr/en` |
| `CommunityRule` | `title_fr/en`, `body_fr/en` |
| `LegalDocument` | `title_fr/en`, `content_fr/en` |
| `Testimonial` | `display_name_fr/en`, `content_fr/en` |
| `WaitingQuestion` | `question_fr/en`, `options_fr/en` |
| `EquipmentType` | `name_fr/en` |
| `CompanionType` | `name_fr/en` |
| `AppNotification` | `title_fr/en`, `body_fr/en` |

### GraphQL — toujours retourner les deux langues

L'API retourne **systématiquement les deux champs** — Flutter choisit selon `User.language` localement, sans refetch :

```graphql
type FaqItem {
  id: ID!
  questionFr: String!
  questionEn: String!
  answerFr: String!
  answerEn: String!
}
```

Flutter bascule instantanément entre les langues sans que l'app ait besoin de redémarrer ni de refetcher.

### `AppNotification` — génération bilingue à la création

Les notifications sont générées dans les deux langues au moment de la création via `TranslationService` :

```python
AppNotification.objects.create(
    user=user,
    type=NotificationType.RUN_CONFIRMED,
    title_fr=translate("notifications", "run_confirmed_title", "fr"),
    title_en=translate("notifications", "run_confirmed_title", "en"),
    body_fr=translate("notifications", "run_confirmed_body", "fr", date=..., neighborhood=...),
    body_en=translate("notifications", "run_confirmed_body", "en", date=..., neighborhood=...),
)
```

### Langue de l'utilisateur — `User.language`

La préférence de langue est persistée sur `User.language` (défaut `fr`) via la mutation `accountsUserUpdateLanguage`.

Le décorateur `@extract_language` utilise cet ordre de priorité :

```
1. Header HTTP `language` (si présent dans la requête)
2. user.language (préférence persistée)
3. "fr" (fallback final)
```

Les messages d'erreur GraphQL utilisent cette langue — Flutter n'a pas à envoyer un header à chaque requête.
